Storing your Life

  • Phase 0: Week 7

Now adays, so much of our life is captured in databases somewhere. Whether it's my sleep and steps in my JawboneUp, my travelling via phone GPS, my preferences, interests, and questions via Reddit or Google- the internet may know me better than some of my friends. But where is all that data going? Into databases, usually. Below are some things to know about the databases that may just be storing your life.

SQL vs. NoSQL

SQL stands for structured query language, and the primary difference between storing your data in SQL databases vs. noSQL databases is exactly that: the way it answers a query. A query, or question, is how one accesses a database. Let's say we're a website and we've collected your data that includes your name, bio, and age. In SQL, that information would be stored in a table form, and you would be one specific user (likely assigned an id number.) If you wanted to access your information from the database, you would send a "query" or question to that table and it would return a set of information.

In NoSQL databases are document based, which means they collect data that don't have a standard schema the way that an SQL database with specific, common inputs has. The site The Geek Stuff has a good set of use/case examples when you would choose NoSQL or SQL depending on what you want longterm for the database.

SQL Injection

One way your data could be threatened is through SQL Injection, a highly common web attack mecahnism used by hackers that allows them to steal data. Hackers inject SQL comands into places like the login form of a website in order to gain access to a database. It is not really a fluc: it happens if the user input fields allows SQL statementsto pass through and query (or ask questions of the data) directly. While a hacker can use this data for their own purposes, they also have the scary ability to delete your data through the SQL command DROP TABLE!

Luckily, chances of this type of attack can be prevented if the programmer is diligent. One way to do this is to put constraints on the types of data that can be entered into an input field. Another is to secure system administrator accounts and remove all stored procedures that are not in use. Being aware of these types of attacks is the first step, and being on top of your work/where your data is going is equally important.
